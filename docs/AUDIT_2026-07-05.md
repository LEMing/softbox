# Architecture Re-Audit — v3.19.0 (2026-07-05)

Fresh A++ re-audit run via an 89-agent multi-perspective workflow (8 dimension
auditors, adversarial per-finding verification — 2 independent refuters for each
high/critical claim). 62 findings confirmed, 7 refuted out of 69 raised.

**Overall grade: A-**

threedviewer landed at a strong A- overall, up from A- three days ago with the full prior 15-item roadmap executed — but the grade didn't move because this cycle's new feature surface (video capture, GLTF animation playback, turntable) introduced its own crop of confirmed high-severity correctness bugs, several verified directly against source in this pass: RenderLoopManager's idle detection never re-arms after releaseContinuous (a permanent no-op rAF loop), turntable+path-tracing can hang captureStill forever, mergeWithPreset shallow-spreads consumer options onto a deep-merged base (confirmed at presets.ts:56), and concurrent captureVideo calls collide on one shared render-loop reason string. The classic architectural dimensions (layer boundaries, disposal, ts discipline, packaging, tests) are all genuinely A/A- territory — core stays provably three-free, disposal funnels are comprehensive, the type/Result discipline is close to spotless, and the consumer-facing packaging gates (attw, dual smoke, type-smoke) are unusually rigorous for a library this size — with the one recurring architectural wart (core's double-cast peek at ThreeRendererAdapter's private field) surfacing identically across four dimension reports. The path to A++ is concrete and mostly small: the RenderLoopManager idle fix and the mergeWithPreset deep-merge are each S/M-sized and eliminate whole classes of the new-surfaces findings, plus the attw FalseESM fix and CI required-checks are process-level closes that lock in what's already built.

## Roadmap to A++ (checklist)

Ordered by priority as synthesized. Check items off as they land.

- [x] **1. [high/M] Fix RenderLoopManager: idle detection never re-arms after releaseContinuous** — PR #53
  Confirmed by reading RenderLoopManager.ts: the idle-timeout-arming code lives inside the `if (shouldRender)` branch. The frame right after releaseContinuous() drops continuousRenderingEnabled to false with needsRender already false, so that frame doesn't render, the idle-arm block is skipped, isIdle never flips true, and shouldContinue stays true forever via `!this.isIdle` — a permanent no-op rAF loop. Move idle-timer arming outside the shouldRender gate (arm it whenever continuousRenderingEnabled is false, independent of whether a render happened), or call requestRender()/arm the timeout directly inside releaseContinuous() when the reason set empties. Add a regression test: captureVideo/pauseAnimations on a staticScene viewer must end with isRunning() === false after idleDelay.

- [x] **2. [high/M] Path tracer self-disable never notifies: releaseContinuous('path-tracing') and captureStill hang** — PR #54
  Emit a 'pathtracing:paused'/'pathtracing:disabled' event on every self-disable path in ThreePathTracingService so the coordinator releases the 'path-tracing' reason; have capturePathTracedStill subscribe to that event (or poll with a timeout) so its promise settles when the tracer gives up rather than hanging indefinitely. Add a unit test per self-disable path.

- [x] **3. [high/S] Guard turntable (autoRotate) + path tracing combination for captureStill** — PR #54
  With autoRotate on, path-traced accumulation is reset every frame, so captureStill()'s wait-for-accumulation never converges and the promise never settles. Either reject with INVALID_STATE when both are active, capture the current accumulation immediately as a documented degraded mode, or temporarily suspend autoRotate for the duration of the capture.

- [x] **4. [high/S] Deep-merge consumer options in mergeWithPreset, not just defaults+preset** — PR #54
  Confirmed in src/presets.ts:56 — `{ ...deepMerge(defaults, resolvePreset(preset)), ...options }` shallow-spreads the caller's options last, so any partial sub-object (e.g. `renderer:{antialias:false}`) wholesale-replaces the deep-merged renderer object, silently dropping ACES tone mapping / preset exposure. Change to `deepMerge(deepMerge(defaults, preset), options)`; this also removes the hand-layering workaround in useResolvedOptions for pathTracing.

- [x] **5. [high/M] Give concurrent captureVideo calls independent render-loop reasons** — PR #56
  Confirmed: both calls use the literal string 'video-capture' as the RenderLoopManager reason (CaptureController.ts:154,173). A second, shorter capture finishing first deletes the shared reason and lets continuous demand drop mid-flight for the first capture, starving its requestFrame forwarding. Use a unique reason per take (`video-capture:${id}`) or reject overlapping captures with INVALID_STATE; add the two-captures-at-once regression test.

- [x] **6. [high/M] Route captureStill/captureVideo through (or reject against) active screenshot-replacement mode** — PR #57
  CaptureController does not check ScreenshotManager state, so calling capture while a screenshot-replacement scene swap is active captures a hidden canvas over a disposed/replaced scene and resolves ok. Add an isScreenshotActive() dependency; either return INVALID_STATE or trigger the ScreenshotManager restore flow first, then capture.

- [x] **7. [high/S] Fix dispose() during in-flight loadModel: orphaned model with no owner to dispose it** — PR #57
  Re-check this.disposed immediately after the await in ViewerCore.runLoadModel; on post-dispose completion, dispose the freshly loaded model and return INVALID_STATE instead of installing it into a torn-down scene. Add the dispose-during-loadModel unit test called out by the tests dimension.

- [x] **8. [high/S] Fix CJS types masquerading as ESM (attw FalseESM) in package.json exports** — PR #55
  The require condition currently serves the ESM .d.ts. Emit a .d.cts twin and split conditions: `"import": {"types": "./dist/index.d.ts", ...}`, `"require": {"types": "./dist/index.d.cts", ...}`. Add `attw --pack` to CI so this class of regression is caught automatically — the in-repo type-smoke gate cannot see it because the resolved TS version (5.9.3) tolerates require(esm).

- [x] **9. [med/S] Replace core's structural double-cast peek at ThreeRendererAdapter's private field with a port method** — PR #53
  Confirmed at ViewerCore.ts:319 — `(this.renderer as unknown as {renderer: null|unknown}).renderer === null`. Add `isDisposed(): boolean` to IRenderer, implement in ThreeRendererAdapter, and use it here. This single fix closes findings duplicated across layer-boundaries, gpu-disposal, ts-error-model and code-health.

- [x] **10. [med/M] Make pauseAnimations actually pause (or fix the documented contract)** — PR #58
  SimpleViewerHandle docs promise resume-from-pose; ThreeAnimationService.play() restarts every clip from t=0. Set action.paused=true/false to implement a true pause/resume, or change the docs and rename the API to reflect restart semantics — a test already exists whose title claims 'resumes' but never asserts it; add the assertion once behavior is decided.

## Confirmed findings (full list, by severity)

### High (10)

- **[layer-boundaries]** Core hardcodes the ThreeRendererAdapter's private field shape, bypassing its own port
  - Location: `src/core/ViewerCore.ts:319`
  - Evidence: if (!this.renderer || (this.renderer as unknown as { renderer: null | unknown }).renderer === null) { this.renderLoopManager.stop();
  - Suggestion: This is a concrete infrastructure-knowledge leak into core: it structurally peeks at ThreeRendererAdapter's private `renderer` field (ThreeRenderer.ts:20 `private renderer: THREE.WebGLRenderer | null`, nulled in dispose() at line 226). The sanctioned escape hatch already exists and returns null after dispose — use `hasInternalRenderer(this.renderer) && this.renderer.getInternalRenderer() === null`, or better, add `isDisposed(): boolean` to IRenderer. As written, any non-Three IRenderer (or a rename of the private field) makes `.renderer` undefined, `undefined === null` is false, and the dead-context guard silently never fires.

- **[gpu-disposal]** Path tracer self-disable paths never notify: continuous render demand leaks forever and capturePathTracedStill can hang indefinitely
  - Location: `src/infrastructure/three/ThreePathTracingService.ts:365`
  - Evidence: Three silent self-disable paths set enabled=false without emitting 'pathtracing:paused': env-wait timeout (line 364-366 'if (this.environmentWaitFrames >= this.maxEnvironmentWaitFrames) { this.enabled = false; ... }'), max create attempts (line 313 'this.enabled = false;'), and PMREM-without-original (line 451 'this.enabled = false;'). The only demand-release paths are PathTracingCoordinator.handleComplete (needs samples >= maxSamples, never reached here) and the 'pathtracing:paused' handler (never fired here). ViewerCore.initialize:227 holds requireContinuous('path-tracing') for staticScene viewers, so the rAF loop renders at full rate forever. Worse, CaptureController.capturePathTracedStill:236 waits on 'pathtracing:complete' while isAccumulating() is true - if the service then self-disables, neither 'pathtracing:complete' nor 'model:error' ever fires and the captureStill promise dangles until dispose.
  - Suggestion: Emit 'pathtracing:paused' (or a dedicated 'pathtracing:disabled' event) on every self-disable path so the coordinator releases the 'path-tracing' reason and unwinds alwaysRender; in capturePathTracedStill, also subscribe to that event (or poll isAccumulating with a timeout) so the promise settles when the tracer gives up. Add a unit test for each self-disable path asserting releaseContinuous is called and captureStill settles.

- **[gpu-disposal]** dispose() during an in-flight loadModel creates resources after teardown with no owner to dispose them
  - Location: `src/core/ViewerCore.ts:283`
  - Evidence: runLoadModel checks this.disposed only at entry; after 'const result = await this.modelManager.loadModel(source, this.events);' (line 283) there is no re-check, and ModelManager.loadModel has none either. If dispose() runs while the GLTF fetch is in flight (routine on React unmount / StrictMode), the resolved load continues: scene.add(model) into the already-disposeContents()'d scene (ModelManager.ts:91), buildRaycastBvh, addDynamicGrid creates fresh grid geometry/materials, attachAnimations builds a new AnimationMixer after dispose already ran detach(), and stateManager.setLoaded fires after setDisposed(). modelManager.dispose() and resourceManager.dispose() have already run, so none of these post-teardown resources are ever disposed.
  - Suggestion: Re-check this.disposed immediately after the await in runLoadModel; on a post-dispose completion, dispose the freshly loaded model (result.value.dispose()) and return INVALID_STATE instead of installing it. Add a test: start loadModel with a deferred loader promise, dispose(), resolve the load, assert the model was disposed and scene.add was never called.

- **[packaging-release]** CJS types masquerade as ESM (attw FalseESM): the require condition serves an ESM .d.ts
  - Location: `package.json:48`
  - Evidence: "exports": { ".": { "types": "./dist/index.d.ts", "import": "./dist/simple-viewer.es.js", "require": "./dist/simple-viewer.cjs" } } — with "type": "module" (line 6), dist/index.d.ts is an ESM declaration file, yet it also types the CJS entry. `npx @arethetypeswrong/cli --pack .` on the actual tarball reports: "node16 (from CJS): 👺 Masquerading as ESM — Import resolved to an ESM type declaration file, but a CommonJS JavaScript file."
  - Suggestion: Emit a .d.cts twin and use per-condition types: ".": { "import": { "types": "./dist/index.d.ts", "default": "./dist/simple-viewer.es.js" }, "require": { "types": "./dist/index.d.cts", "default": "./dist/simple-viewer.cjs" } }. Note the in-repo type-smoke gate cannot see this: the lockfile resolves TypeScript 5.9.3, which permits require(esm) under nodenext, so consumer.cts compiles — but consumers on TS 5.0–5.7 with module node16/nodenext get a hard TS1479 on require('threedviewer') even though the runtime .cjs works. Adding `attw --pack` to CI would lock the fix in.

- **[react]** mergeWithPreset shallow-spreads consumer options, so any partial sub-object wholesale-replaces defaults+preset — e.g. renderer:{antialias:false} silently drops ACES tone mapping and shadow config
  - Location: `src/presets.ts:56`
  - Evidence: `return { ...deepMerge(defaults, resolvePreset(options.preset)), ...options };` — a consumer renderer object replaces the whole merged renderer. Downstream, RendererOptionsConverter only configures tone mapping `if (options.toneMapping !== undefined || options.toneMappingExposure !== undefined)` (RendererOptionsConverter.ts:24), so THREE falls back to NoToneMapping/exposure 1 instead of the library's ACES 6/1.1. It also breaks the live path: with preset+partial renderer, useViewerCore's runtime effect sends `toneMappingExposure: undefined`, so preset switches change background but never exposure. README line 184 promises "Pass options to override any part".
  - Suggestion: Deep-merge consumer options too (`deepMerge(deepMerge(defaults, preset), options)`) — deepMerge already ignores undefined and replaces arrays whole. The pathTraced-shorthand workaround in useResolvedOptions (which hand-layers defaultOptions.pathTracing) becomes unnecessary. If wholesale replacement is truly intended, document it prominently instead; today the README and the preset deep-merge behaviour imply field-level override.

- **[tests]** Concurrent captureVideo calls collide on the shared 'video-capture' reason — second capture loses its render-loop hold; the exact edge case is untested
  - Location: `src/core/CaptureController.ts:154`
  - Evidence: CaptureController.ts:154 `this.deps.renderLoopManager.requireContinuous('video-capture')` and finish() at :173 `releaseContinuous('video-capture')`; RenderLoopManager.ts:59-62 backs reasons with a plain Set<string> (`this.continuousReasons.delete(reason); this.continuousRenderingEnabled = this.continuousReasons.size > 0;`), so two overlapping captures share one Set entry. CaptureControllerVideo.test.ts has zero concurrent-capture tests (grep for concurrent/two captures: no hits) and no guard rejects a second capture.
  - Suggestion: On a static scene (idle detection on), start capture A (duration 10) then capture B (duration 2): when B finishes it deletes the single 'video-capture' entry, continuous demand drops, the loop idles out ~1s later, render:complete stops firing, and A's requestFrame forwarding starves — A records frozen/short footage for its remaining 7s. Use a unique reason per take (e.g. `video-capture#${id}`) or reject concurrent captures with INVALID_STATE, and add the two-captures-at-once test.

- **[tests]** Test title claims 'play() resumes' after pause but never tests it — and the implementation actually restarts from t=0, not resumes
  - Location: `src/infrastructure/three/__tests__/ThreeAnimationService.test.ts:51`
  - Evidence: The test 'pause() freezes the pose and play() resumes' (lines 51-62) pauses and asserts the frozen pose but never calls play() again, so 'resumes' is unverified. The implementation would fail that assertion: play() at ThreeAnimationService.ts:44 calls `this.mixer.stopAllAction();` which resets every action's time to 0, so play() after pause() snaps the pose back to the clip start instead of resuming.
  - Suggestion: Add the missing resume assertion (play() after pause(), update(0.25), expect x≈5 if resuming). It will fail — then decide the contract: either skip stopAllAction when re-playing the same clip set (true resume) or rename the API/docs to 'restart' so the docstring, test name, and behavior agree.

- **[new-surfaces]** releaseContinuous never re-arms idle detection: captureVideo and pauseAnimations leave a permanent no-op rAF loop on staticScene viewers
  - Location: `src/core/utils/RenderLoopManager.ts:59`
  - Evidence: releaseContinuous(reason) only does `this.continuousReasons.delete(reason); this.continuousRenderingEnabled = this.continuousReasons.size > 0;` — the idle timeout is armed exclusively inside the shouldRender branch (lines 126-129), which never executes again once shouldRender is false, while shouldContinue stays true because isIdle is never set. The unit test works around exactly this: `manager.releaseContinuous('turntable'); manager.requestRender(); // the final frame arms the idle timer` (RenderLoopManager.test.ts:63-64). CaptureController.finish() (CaptureController.ts:173) and ViewerCore.pauseAnimations() (ViewerCore.ts:598) release without any follow-up requestRender or stop.
  - Suggestion: In releaseContinuous, when the reason set becomes empty, call requestRender() (one frame renders with continuous off and arms the idle timer) or arm the idle timeout directly. Add a regression test: captureVideo/pauseAnimations on a staticScene viewer must end with isRunning() === false after idleDelay.

- **[new-surfaces]** turntable + pathTraced together make captureStill() a promise that never settles and reset the accumulation forever
  - Location: `src/core/ViewerCore.ts:327`
  - Evidence: With autoRotate on, controls.update() reports a change every frame, so the loop runs `this.pathTracing.resetAccumulation()` (ViewerCore.ts:331) every frame; ThreePathTracingService.reset() throttles to one reset per 50ms (ThreePathTracingService.ts:628), so sampleCount can never reach DEFAULT_PATH_TRACING_SAMPLES = 300 and 'pathtracing:complete' never fires. capturePathTracedStill then waits on exactly that event (CaptureController.ts:236-248) with no timeout — the public captureStill() promise hangs until dispose, while the tracer burns GPU on 1-3-sample frames indefinitely.
  - Suggestion: Guard the combination: when autoRotate is enabled with path tracing, either capture the current accumulation immediately (documented degraded mode), reject with INVALID_STATE, or temporarily suspend autoRotate for the capture. Separately consider pausing PT entirely while a continuous 'turntable' reason is held — accumulate-and-reset every frame is pure waste.

- **[new-surfaces]** captureStill/captureVideo ignore screenshot-replacement mode: they capture a hidden canvas over a disposed scene and resolve ok
  - Location: `src/core/CaptureController.ts:97`
  - Evidence: With replaceWithScreenshotOnComplete, ScreenshotManager sets `canvas.style.display = 'none'` (ScreenshotManager.ts:78) and the model/scene resources are disposed (ViewerCore.replaceWithScreenshot → disposeCurrentModel + disposeSceneResources). CaptureController never consults screenshotManager.isActive(): captureRasterStill re-renders the now-empty scene into the hidden canvas and returns Result.ok of a blank frame, and captureVideo records the whole duration while ViewerCore.renderFrame early-returns (`if (this.screenshotManager.isActive()) return;`, ViewerCore.ts:384-386), so no render:complete/requestFrame ever fires and an empty/frozen clip resolves as success.
  - Suggestion: Give CaptureController an isScreenshotActive() dep. Either return INVALID_STATE, or (better UX) trigger the ScreenshotManager restore flow, await the reload, capture, and let the screenshot replacement re-engage.

- **[gpu-disposal]** WebGL contexts are never proactively released: no forceContextLoss on renderer dispose and isSupported() leaks a probe context
  - Location: `src/infrastructure/three/ThreeRenderer.ts:225`
