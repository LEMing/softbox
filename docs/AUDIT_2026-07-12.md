## Confirmed highs (adversarially verified, 4/4)

### H1. Composer passes never disposed â†’ repeated VRAM leak on every runtime effects change
`PostProcessingPipeline.dispose()` (src/infrastructure/three/postprocessing/PostProcessingPipeline.ts:268)
relies on `EffectComposer.dispose()`, which in three 0.185.1 frees only its two
ping-pong targets + copyPass â€” never user-added passes. UnrealBloomPass holds
11 HalfFloat render targets + 8 materials with its own `dispose()` that nothing
calls. `ThreeRenderer.setPostProcessing` disposes-and-rebuilds the pipeline on
every effects change (ViewerCore.updateOptions â†’ postEffectsKey diff), so with
bloom active each toggle strands **â‰ˆ61 MB** on a 1920Ă—1080 DPR-2 canvas; a
dozen toggles â‰ˆ 0.6 GB â†’ context loss on mobile Safari / iGPUs, with no restore
handler. Non-bloom toggles leak only materials (near-negligible).
**Fix (S):** track added passes; dispose each in `PostProcessingPipeline.dispose()`.

### H2. Second HDRI aliases to the first: background, PT lighting, and studio reset all pick the wrong texture
`ThreeEnvironmentService.applyToScene` resolves the "original" equirect via
`find(([key, tex]) => key.endsWith('_original') && tex.mapping === Equirectangularâ€¦)`
over the whole cache (src/infrastructure/three/ThreeEnvironmentService.ts:215-217,
243-245) â€” every `_original` has that mapping, the Map never shrinks, so the
FIRST-loaded HDRI always wins. After `setEnvironmentMap(A)` then `(B)`:
reflections show B, backdrop shows A, and the tracer re-ingests A's lighting.
Corollary: `resetEnvironment()` after any HDRI takes the `originalEntry` branch
before the studio-cube branch (:219-227), so PT keeps old-HDRI lighting while
raster shows studio. No test loads two HDRIs sequentially.
**Fix (S/M):** key the lookup by `url + '_original'` (the URL is in hand at
`setEnvironmentMap`); reorder/guard the studio-cube branch; add the two-HDRI test.

### H3. Resize during PT accumulation desyncs three state machines at once
The resize path resets nothing unless the frame already converged
(`PathTracingCoordinator.onResizeStart` guards on `completeHandled`,
src/core/PathTracingCoordinator.ts:348-355). Mid-accumulation resize:
**(a)** tracer camera goes stale â€” `updateCamera()` fires only on `cameraDirty`,
set only by `reset()`; traced frames render aspect-distorted until the next
camera move, presented at FULL opacity (the `_updateScale` reset skips the
opacity ramp). **(b)** counter divergence â€” the tracer's internal
`setSize â†’ reset()` zeroes ITS samples while the service's `sampleCount` keeps
climbing; completion fires with far fewer real samples, freezing a noisy frame
as "final" and settling `captureStill` on noise. **(c)** the dissolve snapshot
is captured only at sample 0; a mid-fade resize composites the old-size
snapshot stretched full-screen at up to 100% opacity.
Inverse asymmetry (medium, F4): for a CONVERGED frame resize resets but never
re-enables (`wasActive` false after self-pause) â€” the high-quality frame is
replaced by raster and PT stays dormant until a camera move.
**Fix (M):** treat resize as a camera move â€” force `resetAccumulation()` when
accumulating (fixes a+b+c in one move); ideally track completion off
`pathTracer.samples`; make the converged branch re-enable.

### H4. `floorAlignment` missing from BOTH option key lists â€” the `units` bug class recurred
