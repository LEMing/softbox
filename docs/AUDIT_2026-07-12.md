## Confirmed highs (adversarially verified, 4/4)

### H1. Composer passes never disposed → repeated VRAM leak on every runtime effects change
`PostProcessingPipeline.dispose()` (src/infrastructure/three/postprocessing/PostProcessingPipeline.ts:268)
relies on `EffectComposer.dispose()`, which in three 0.185.1 frees only its two
ping-pong targets + copyPass — never user-added passes. UnrealBloomPass holds
11 HalfFloat render targets + 8 materials with its own `dispose()` that nothing
calls. `ThreeRenderer.setPostProcessing` disposes-and-rebuilds the pipeline on
every effects change (ViewerCore.updateOptions → postEffectsKey diff), so with
bloom active each toggle strands **≈61 MB** on a 1920×1080 DPR-2 canvas; a
dozen toggles ≈ 0.6 GB → context loss on mobile Safari / iGPUs, with no restore
handler. Non-bloom toggles leak only materials (near-negligible).
**Fix (S):** track added passes; dispose each in `PostProcessingPipeline.dispose()`.

### H2. Second HDRI aliases to the first: background, PT lighting, and studio reset all pick the wrong texture
`ThreeEnvironmentService.applyToScene` resolves the "original" equirect via
`find(([key, tex]) => key.endsWith('_original') && tex.mapping === Equirectangular…)`
over the whole cache (src/infrastructure/three/ThreeEnvironmentService.ts:215-217,
243-245) — every `_original` has that mapping, the Map never shrinks, so the
FIRST-loaded HDRI always wins. After `setEnvironmentMap(A)` then `(B)`:
reflections show B, backdrop shows A, and the tracer re-ingests A's lighting.
Corollary: `resetEnvironment()` after any HDRI takes the `originalEntry` branch
before the studio-cube branch (:219-227), so PT keeps old-HDRI lighting while
raster shows studio. No test loads two HDRIs sequentially.
**Fix (S/M):** key the lookup by `url + '_original'` (the URL is in hand at
`setEnvironmentMap`); reorder/guard the studio-cube branch; add the two-HDRI test.

### H3. Resize during PT accumulation desyncs three state machines at once
The resize path resets nothing unless the frame already converged
(`PathTracingCoordinator.onResizeStart` guards on `completeHandled`,
src/core/PathTracingCoordinator.ts:348-355). Mid-accumulation resize:
**(a)** tracer camera goes stale — `updateCamera()` fires only on `cameraDirty`,
set only by `reset()`; traced frames render aspect-distorted until the next
camera move, presented at FULL opacity (the `_updateScale` reset skips the
opacity ramp). **(b)** counter divergence — the tracer's internal
`setSize → reset()` zeroes ITS samples while the service's `sampleCount` keeps
climbing; completion fires with far fewer real samples, freezing a noisy frame
as "final" and settling `captureStill` on noise. **(c)** the dissolve snapshot
is captured only at sample 0; a mid-fade resize composites the old-size
snapshot stretched full-screen at up to 100% opacity.
Inverse asymmetry (medium, F4): for a CONVERGED frame resize resets but never
re-enables (`wasActive` false after self-pause) — the high-quality frame is
replaced by raster and PT stays dormant until a camera move.
**Fix (M):** treat resize as a camera move — force `resetAccumulation()` when
accumulating (fixes a+b+c in one move); ideally track completion off
`pathTracer.samples`; make the converged branch re-enable.

### H4. `floorAlignment` missing from BOTH option key lists — the `units` bug class recurred
Absent from the structural key (useStableOptions.ts:50-96) and the runtime set
(runtimeOptions.ts): changing it on a mounted viewer is a silent no-op — and
worse, the dropped value suddenly applies later, as a full rebuild with a
visible model jump, when any UNRELATED structural option changes
(stableOptionsRef only recomputes on key change). It is a construction-time
ModelManager input (ModelManager.ts:36,47) exactly like `units`, which IS in
the structural key with a comment. Severity medium-high (niche option), but
this is the second recurrence of the class — the real fix is the guard below.
**Fix (S):** add to the structural stringify + deps; **plus** an exhaustiveness
test partitioning `keyof SimpleViewerOptions` into structural ∪ runtime ∪
explicitly-excluded, so the third recurrence is unrepresentable.

## Mediums (selected, confirmed by dimension auditors)

- **M1** (ts-error F1): runtime PT enable → lazy-chunk load failure is an
  **unhandled promise rejection** — `LazyPathTracingService.ts:31` await outside
  try/catch, coordinator try/finally without catch, `ViewerCore.ts:788`
  `.then()` without `.catch()`. The only uncovered async boundary in the codebase.
- **M2** (ts-error F2/F3): effects-chunk load failure and PT init failure die in
  `console.warn` — never reach the consumer's `'error'` event; programmatically
  indistinguishable from success.
- **M3** (new-surfaces F3): runtime `backgroundColor` / `environmentIntensity`
  changes repaint raster but never force a PT re-ingest — a converged traced
  frame keeps the old look indefinitely (preset switch during PT).
- **M4** (disposal): `resetEnvironment()` pushes a new PMREM target per call,
  unbounded within a session; `convertedEnvTexture` cache is keyed to nothing
  and goes stale after a runtime env switch (retains the old float texture).
- **M5** (react): `helpers.gizmo` — pure DOM chrome — sits in the structural key:
  toggling it tears down the WebGL context and refetches the model.
- **M6** (new-surfaces F5): `staticScene: false` viewers permanently lose
  `alwaysRender` after the first PT pause; external scene mutations stop
  repainting.
- **M7** (tests): global ratchet debt ~10 pts; dissolve resource reuse
  unasserted (allocate-once guard removable without a test failing).
